# <a name="ui-automation-and-active-accessibility"></a>Automação e Acessibilidade Ativa da interface do usuário

o Microsoft Acessibilidade Ativa é a API herdada introduzida no Windows 95 e foi projetada para tornar os aplicativos Windows acessíveis. a automação da interface do usuário da Microsoft é o novo modelo de acessibilidade para Windows e destina-se a atender às necessidades de produtos de tecnologia assistencial e ferramentas de teste automatizadas. A automação da interface do usuário oferece muitas melhorias em relação ao Microsoft Acessibilidade Ativa. Este tópico explica as diferenças entre as duas tecnologias.

## <a name="programming-languages"></a>Linguagens de programação

O Microsoft Acessibilidade Ativa é baseado no Component Object Model (COM) com suporte para interfaces duplas e, portanto, é programável em C/C++ e em linguagens de script.

Quando a automação da interface do usuário foi introduzida, a API do cliente era limitada a código gerenciado, enquanto a API do provedor incluía implementações gerenciadas e não gerenciadas. com o Windows 7, foi introduzida uma nova API de cliente com base em com para facilitar o programa de aplicativos cliente de automação da interface do usuário em C/C++.

## <a name="servers-and-clients"></a>Servidores e clientes

No Microsoft Acessibilidade Ativa, os servidores e os clientes se comunicam diretamente, principalmente por meio da implementação do servidor da interface [**IAccessible**](/windows/desktop/api/oleacc/nn-oleacc-iaccessible) .

Na automação da interface do usuário, um serviço principal está entre o servidor (provedor) e o cliente. O serviço principal faz chamadas para as interfaces implementadas pelos provedores e fornece serviços adicionais, como a geração de identificadores exclusivos de tempo de execução para elementos da interface do usuário. Os aplicativos cliente têm acesso a esse serviço principal criando um objeto [**CUIAutomation**](/previous-versions/windows/desktop/legacy/ff384838(v=vs.85)) . Esse objeto dá suporte a um conjunto de interfaces de cliente que são separadas das interfaces do provedor. Para obter mais informações, consulte [criando o objeto CUIAutomation](uiauto-creatingcuiautomation.md).

Provedores de automação de interface do usuário podem fornecer informações para clientes Microsoft Acessibilidade Ativa e os servidores Microsoft Acessibilidade Ativa podem fornecer informações para aplicativos cliente de automação da interface do usuário. No entanto, como o Microsoft Acessibilidade Ativa não expõe tantas informações quanto a automação da interface do usuário, os dois modelos não são totalmente compatíveis.

## <a name="ui-elements"></a>Elementos da interface do usuário

O Microsoft Acessibilidade Ativa apresenta um elemento de interface do usuário como uma interface [**IAccessible**](/windows/desktop/api/oleacc/nn-oleacc-iaccessible) emparelhada com um identificador filho. É difícil comparar dois ponteiros **IAccessible** para determinar se eles se referem ao mesmo elemento.

Na automação da interface do usuário, cada elemento é representado como um objeto que expõe a interface [**IUIAutomationElement**](/windows/desktop/api/UIAutomationClient/nn-uiautomationclient-iuiautomationelement) aos clientes. Os elementos podem ser comparados por seus identificadores de tempo de execução, que são recuperados usando [**IUIAutomationElement:: GetRuntimeId**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-getruntimeid).

## <a name="tree-views-and-navigation"></a>Exibições de árvore e navegação

Os elementos da interface do usuário na tela podem ser vistos como uma estrutura de árvore com a área de trabalho como raiz, janelas de aplicativo como filhos imediatos e elementos dentro de aplicativos como mais descendentes.

No Microsoft Acessibilidade Ativa, muitos elementos da interface do usuário que são irrelevantes para os usuários finais são expostos na estrutura de árvore. Os aplicativos cliente devem examinar todos os elementos na árvore para determinar quais elementos são significativos.

Aplicativos cliente de automação da interface do usuário veem a interface do usuário por meio de uma exibição filtrada A exibição contém apenas elementos que fornecem informações ao usuário ou com os quais o usuário pode interagir. Exibições predefinidas que incluem apenas elementos de controle e apenas elementos de conteúdo estão disponíveis, e aplicativos cliente podem definir modos de exibição personalizados. A automação da interface do usuário torna mais fácil descrever a interface do usuário e ajudar o usuário a interagir com aplicativos.

No Microsoft Acessibilidade Ativa, a navegação entre os elementos é espacial, por exemplo, movendo para o elemento que está à esquerda na tela, lógico, por exemplo, movendo para o próximo item de menu ou o próximo item na ordem de tabulação em uma caixa de diálogo ou hierárquica, por exemplo, movendo para o primeiro elemento filho em um contêiner ou de um elemento filho para seu elemento pai. A navegação hierárquica é complicada pelo fato de que os elementos filho nem sempre são objetos que implementam o [**IAccessible**](/windows/desktop/api/oleacc/nn-oleacc-iaccessible).

Na automação da interface do usuário, todos os elementos da interface do usuário são objetos COM que expõem a interface [**IUIAutomationElement**](/windows/desktop/api/UIAutomationClient/nn-uiautomationclient-iuiautomationelement) e dão suporte à mesma funcionalidade básica. Do ponto de vista do provedor, os objetos COM implementam uma interface que é herdada de [**IRawElementProviderSimple**](/windows/desktop/api/UIAutomationCore/nn-uiautomationcore-irawelementprovidersimple). A navegação é principalmente hierárquica; ou seja, de pais a filhos e de um irmão para o outro. No entanto, a navegação entre irmãos tem um elemento lógico, pois ele pode seguir a ordem de tabulação. Um cliente pode navegar de qualquer ponto de partida, usando qualquer exibição filtrada da árvore, usando [**IUIAutomationTreeWalker**](/windows/desktop/api/UIAutomationClient/nn-uiautomationclient-iuiautomationtreewalker). Um cliente também pode navegar para filhos ou descendentes específicos usando [**IUIAutomationElement:: FindFirst**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-findfirst) e [**IUIAutomationElement:: FindAll**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-findall). Por exemplo, é fácil recuperar todos os elementos em uma caixa de diálogo que dão suporte a um padrão de controle especificado.

A navegação na automação da interface do usuário é mais consistente do que no Microsoft Acessibilidade Ativa. Alguns elementos, como listas suspensas e janelas pop-up, aparecem duas vezes na árvore do Microsoft Acessibilidade Ativa, e a navegação desses elementos pode ter resultados inesperados. É difícil implementar o Microsoft Acessibilidade Ativa corretamente para um controle rebar. A automação da interface do usuário permite a recolocação e o reposicionamento, para que um elemento possa ser colocado em qualquer lugar da árvore, apesar da hierarquia imposta pela propriedade do Windows.

## <a name="roles-and-control-types"></a>Funções e tipos de controle

O Microsoft Acessibilidade Ativa usa a propriedade accRole ([**IAccessible:: get \_ accRole**](/windows/desktop/api/Oleacc/nf-oleacc-iaccessible-get_accrole)) para recuperar uma descrição da função do elemento na interface do usuário, [**como \_ \_ controle deslizante do sistema de função**](object-roles.md) ou sistema de [**função \_ \_ MenuItem**](object-roles.md). A função de um elemento é a principal pista para sua funcionalidade disponível. A interação com um controle é obtida usando métodos fixos como [**IAccessible:: accSelect**](/windows/desktop/api/Oleacc/nf-oleacc-iaccessible-accselect) e [**IAccessible:: accDoDefaultAction**](/windows/desktop/api/Oleacc/nf-oleacc-iaccessible-accdodefaultaction). A interação entre o aplicativo cliente e a interface do usuário é limitada ao que pode ser feito por meio do [**IAccessible**](/windows/desktop/api/oleacc/nn-oleacc-iaccessible).

Por outro lado, a automação da interface do usuário dissocia o tipo de controle do elemento, que é descrito pela propriedade [**IUIAutomationElement:: CurrentControlType**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-get_currentcontroltype) (ou [**IUIAutomationElement:: CachedControlType**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-get_cachedcontroltype)), de sua funcionalidade esperada. A funcionalidade é determinada pelos padrões de controle que são suportados pelo provedor por meio de sua implementação de interfaces especializadas. Os padrões de controle podem ser combinados para descrever o conjunto completo de funcionalidades com suporte em um determinado elemento de interface do usuário. Alguns provedores são necessários para dar suporte a um padrão de controle específico. Por exemplo, o provedor de uma caixa de seleção deve oferecer suporte ao padrão de controle de [alternância](uiauto-implementingtoggle.md) . Outros provedores são necessários para dar suporte a um ou mais de um conjunto de padrões de controle. Por exemplo, um botão deve dar suporte ao padrão de controle Toggle ou [Invoke](uiauto-implementinginvoke.md) . Outros ainda não dão suporte a nenhum padrão de controle. Por exemplo, um painel que não pode ser movido, redimensionado ou encaixado não tem padrões de controle.

A automação da interface do usuário dá suporte a controles personalizados, que são identificados pela constante [**UIA \_ CustomControlTypeId**](uiauto-controltype-ids.md) e podem ser descritos pela propriedade [**IUIAutomationElement:: CurrentLocalizedControlType**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-get_currentlocalizedcontroltype) (ou [**IUIAutomationElement:: CachedLocalizedControlType**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-get_cachedlocalizedcontroltype)).

## <a name="states-and-properties"></a>Estados e propriedades

Os elementos do Microsoft Acessibilidade Ativa dão suporte a um conjunto comum de propriedades. Algumas propriedades, como accState, devem descrever condições diferentes, dependendo da função do elemento. Os servidores devem implementar todos os métodos de [**IAccessible**](/windows/desktop/api/oleacc/nn-oleacc-iaccessible) que retornam uma propriedade, mesmo as propriedades que não são relevantes para o elemento.

A automação da interface do usuário define propriedades adicionais, algumas das quais correspondem a Estados no Microsoft Acessibilidade Ativa. Algumas propriedades são comuns a todos os elementos, mas outras propriedades são específicas para tipos de controle e padrões de controle. Um provedor de automação de interface do usuário não precisa implementar propriedades irrelevantes, mas pode retornar um valor nulo para todas as propriedades para as quais não há suporte. O serviço de núcleo de automação de interface do usuário pode obter algumas propriedades do provedor de janela padrão, e elas são mesclados com propriedades explicitamente implementadas pelo provedor.

Além de dar suporte a muitas outras propriedades, a automação da interface do usuário permite um melhor desempenho, permitindo que as propriedades sejam armazenadas em cache.

## <a name="events"></a>Eventos

Ao contrário Microsoft Active Accessibility, o mecanismo de evento no Automação da Interface do Usuário não depende do roteamento de eventos do Windows, que está intimamente vinculado a alças de janela e não exige que o aplicativo cliente configurar ganchos. As assinaturas de eventos podem ser ajustadas a determinadas partes da árvore, não apenas a eventos específicos. Os provedores também podem ajustar os eventos de adoção, mantendo o controle de quais eventos estão sendo escutados.

Também é mais fácil para os clientes recuperarem os elementos que aumentem eventos porque eles são passados diretamente para o retorno de chamada de evento. As propriedades do elemento são pré-buscadas automaticamente, se uma solicitação de cache foi fornecida quando o cliente assinou o evento.

## <a name="accessing-active-accessibility-properties-and-objects-from-ui-automation"></a>Acessando Propriedades de Acessibilidade Ativa e objetos da automação da interface do usuário

Um recurso importante da automação da interface do usuário que não está disponível no Microsoft Acessibilidade Ativa é a capacidade de buscar várias propriedades com uma única operação de processamento cruzado.

Os clientes existentes do Microsoft Acessibilidade Ativa podem aproveitar essa capacidade usando a interface [**IUIAutomationLegacyIAccessiblePattern**](/windows/desktop/api/UIAutomationClient/nn-uiautomationclient-iuiautomationlegacyiaccessiblepattern) . Essa interface representa um *padrão de controle* que expõe as propriedades e métodos do Microsoft acessibilidade ativa em elementos de interface do usuário. Ao recuperar elementos, um aplicativo pode solicitar que esse padrão de controle e suas propriedades sejam armazenados em cache.

O [**IUIAutomationLegacyIAccessiblePattern**](/windows/desktop/api/UIAutomationClient/nn-uiautomationclient-iuiautomationlegacyiaccessiblepattern) também permite que os clientes obtenham Propriedades do Microsoft acessibilidade ativa de elementos que não têm suporte nativo para o [**IAccessible**](/windows/desktop/api/oleacc/nn-oleacc-iaccessible).

As alterações nas propriedades de um [**IUIAutomationLegacyIAccessiblePattern**](/windows/desktop/api/UIAutomationClient/nn-uiautomationclient-iuiautomationlegacyiaccessiblepattern) não geram eventos de automação da interface do usuário.
