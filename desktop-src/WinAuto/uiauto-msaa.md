# <a name="ui-automation-and-active-accessibility"></a>Automação da Interface do Usuário e Acessibilidade Ativa

Microsoft Active Accessibility é a API herdada que foi introduzida no Windows 95 e foi projetada para tornar Windows aplicativos acessíveis. O Microsoft Automação da Interface do Usuário é o novo modelo de acessibilidade para Windows e destina-se a atender às necessidades de produtos de tecnologia adaptativa e ferramentas de teste automatizado. Automação da Interface do Usuário oferece muitas melhorias em relação Microsoft Active Accessibility. Este tópico explica as diferenças entre as duas tecnologias.

## <a name="programming-languages"></a>Linguagens de programação

Microsoft Active Accessibility é baseado no COM (Component Object Model) com suporte para interfaces duplas e, portanto, é programável em C/C++ e linguagens de script.

Quando Automação da Interface do Usuário foi introduzida, a API do cliente era limitada ao código gerenciado, enquanto a API do provedor incluía implementações gerenciadas e não gerenciadas. Com Windows 7, uma nova API de cliente baseada em COM foi introduzida para facilitar o programar Automação da Interface do Usuário aplicativos cliente em C/C++.

## <a name="servers-and-clients"></a>Servidores e clientes

No Microsoft Active Accessibility, os servidores e os clientes se comunicam diretamente, em grande parte por meio da implementação do servidor da interface [**IAccessible.**](/windows/desktop/api/oleacc/nn-oleacc-iaccessible)

No Automação da Interface do Usuário, um serviço principal está entre o servidor (provedor) e o cliente. O serviço principal faz chamadas para as interfaces implementadas por provedores e fornece serviços adicionais, como gerar identificadores de tempo de execução exclusivos para elementos de interface do usuário. Os aplicativos cliente têm acesso a esse serviço principal criando um [**objeto CUIAutomation.**](/previous-versions/windows/desktop/legacy/ff384838(v=vs.85)) Esse objeto dá suporte a um conjunto de interfaces de cliente que são separadas das interfaces do provedor. Para obter mais informações, [consulte Criando o objeto CUIAutomation](uiauto-creatingcuiautomation.md).

Automação da Interface do Usuário provedores podem fornecer informações para clientes Microsoft Active Accessibility e Microsoft Active Accessibility servidores podem fornecer informações para Automação da Interface do Usuário aplicativos cliente. No entanto, como Microsoft Active Accessibility não expõe tanto informações quanto Automação da Interface do Usuário, os dois modelos não são totalmente compatíveis.

## <a name="ui-elements"></a>Elementos da interface do usuário

Microsoft Active Accessibility apresenta um elemento de interface do usuário como uma interface [**IAccessible**](/windows/desktop/api/oleacc/nn-oleacc-iaccessible) emparelhada com um identificador filho. É difícil comparar dois **ponteiros IAccessible** para determinar se eles se referem ao mesmo elemento.

No Automação da Interface do Usuário, cada elemento é representado como um objeto que expõe a interface [**IUIAutomationElement**](/windows/desktop/api/UIAutomationClient/nn-uiautomationclient-iuiautomationelement) aos clientes. Os elementos podem ser comparados por seus identificadores de tempo de execução, que são recuperados usando [**IUIAutomationElement::GetRuntimeId**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-getruntimeid).

## <a name="tree-views-and-navigation"></a>Exibições de árvore e navegação

Os elementos da interface do usuário na tela podem ser vistos como uma estrutura de árvore com a área de trabalho como raiz, janelas de aplicativo como filhos imediatos e elementos dentro de aplicativos como descendentes posteriores.

No Microsoft Active Accessibility, muitos elementos da interface do usuário que são irrelevantes para os usuários finais são expostos na estrutura de árvore. Os aplicativos cliente devem examinar todos os elementos na árvore para determinar quais elementos são significativos.

Automação da Interface do Usuário aplicativos cliente veem a interface do usuário por meio de uma exibição filtrada. A exibição contém apenas elementos que dão informações ao usuário ou com os que o usuário pode interagir. Exibições predefinidos que incluem apenas elementos de controle e apenas elementos de conteúdo estão disponíveis, e os aplicativos cliente podem definir exibições personalizadas. Automação da Interface do Usuário torna mais fácil descrever a interface do usuário para o usuário e ajudar o usuário a interagir com aplicativos.

No Microsoft Active Accessibility, a navegação entre elementos é espacial, por exemplo, mover para o elemento que está à esquerda na tela, lógico, por exemplo, mover para o próximo item de menu ou o próximo item na ordem de tabulação em uma caixa de diálogo ou hierárquico, por exemplo, mover para o primeiro elemento filho em um contêiner ou de um elemento filho para seu elemento pai. A navegação hierárquica é complicada pelo fato de que os elementos filho nem sempre são objetos que implementam [**IAccessible.**](/windows/desktop/api/oleacc/nn-oleacc-iaccessible)

No Automação da Interface do Usuário, todos os elementos da interface do usuário são objetos COM que expõem a interface [**IUIAutomationElement**](/windows/desktop/api/UIAutomationClient/nn-uiautomationclient-iuiautomationelement) e suportam a mesma funcionalidade básica. Do ponto de vista do provedor, os objetos COM implementam uma interface herdada de [**IRawElementProviderSimple.**](/windows/desktop/api/UIAutomationCore/nn-uiautomationcore-irawelementprovidersimple) A navegação é principalmente hierárquica; ou seja, de pais a filhos e de um irmão para o próximo. No entanto, a navegação entre irmãos tem um elemento lógico, pois ele pode seguir a ordem de tabulação. Um cliente pode navegar de qualquer ponto de partida, usando qualquer exibição filtrada da árvore, usando [**IUIAutomationTreeWalker**](/windows/desktop/api/UIAutomationClient/nn-uiautomationclient-iuiautomationtreewalker). Um cliente também pode navegar para filhos ou descendentes específicos usando [**IUIAutomationElement::FindFirst**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-findfirst) [**e IUIAutomationElement::FindAll**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-findall). Por exemplo, é fácil recuperar todos os elementos em uma caixa de diálogo que suportam um padrão de controle especificado.

A navegação Automação da Interface do Usuário é mais consistente do que no Microsoft Active Accessibility. Alguns elementos, como listas listadas e janelas pop-up, aparecem duas vezes na árvore Microsoft Active Accessibility e a navegação desses elementos pode ter resultados inesperados. É difícil implementar o Microsoft Active Accessibility corretamente para um controle de barra de rebar. Automação da Interface do Usuário permite o reposicionamento e o reposicionamento, de modo que um elemento possa ser colocado em qualquer lugar na árvore, apesar da hierarquia imposta pela propriedade das janelas.

## <a name="roles-and-control-types"></a>Funções e tipos de controle

Microsoft Active Accessibility usa a propriedade accRole ([**IAccessible::get \_ accRole**](/windows/desktop/api/Oleacc/nf-oleacc-iaccessible-get_accrole)) para recuperar uma descrição da função de elemento na interface do usuário, como [**ROLE SYSTEM \_ \_ SLIDER**](object-roles.md) ou [**ROLE SYSTEM \_ \_ MENUITEM**](object-roles.md). A função de um elemento é a principal dica para sua funcionalidade disponível. A interação com um controle é feita usando métodos fixos como [**IAccessible::accSelect**](/windows/desktop/api/Oleacc/nf-oleacc-iaccessible-accselect) e [**IAccessible::accDoDefaultAction**](/windows/desktop/api/Oleacc/nf-oleacc-iaccessible-accdodefaultaction). A interação entre o aplicativo cliente e a interface do usuário é limitada ao que pode ser feito por meio de [**IAccessible.**](/windows/desktop/api/oleacc/nn-oleacc-iaccessible)

Por outro lado, Automação da Interface do Usuário desajusta o tipo de controle do elemento, que é descrito pela propriedade [**IUIAutomationElement::CurrentControlType**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-get_currentcontroltype) (ou [**IUIAutomationElement::CachedControlType**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-get_cachedcontroltype)), de sua funcionalidade esperada. A funcionalidade é determinada pelos padrões de controle com suporte do provedor por meio de sua implementação de interfaces especializadas. Os padrões de controle podem ser combinados para descrever o conjunto completo de funcionalidades com suporte por um elemento de interface do usuário específico. Alguns provedores são necessários para dar suporte a um padrão de controle específico. Por exemplo, o provedor de uma caixa de seleção deve dar suporte ao padrão [de controle Alternar.](uiauto-implementingtoggle.md) Outros provedores são necessários para dar suporte a um ou mais de um conjunto de padrões de controle. Por exemplo, um botão deve dar suporte à alternância ou ao [padrão de controle Invoke.](uiauto-implementinginvoke.md) Ainda assim, outros não suportam nenhum padrão de controle. Por exemplo, um painel que não pode ser movido, relizado ou encaixado não tem padrões de controle.

Automação da Interface do Usuário dá suporte a controles personalizados, que são identificados pela constante [**\_ CustomControlTypeId**](uiauto-controltype-ids.md) da UIA e podem ser descritos pela propriedade [**IUIAutomationElement::CurrentLocalizedControlType**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-get_currentlocalizedcontroltype) [**(ou IUIAutomationElement::CachedLocalizedControlType**](/windows/desktop/api/UIAutomationClient/nf-uiautomationclient-iuiautomationelement-get_cachedlocalizedcontroltype)).

## <a name="states-and-properties"></a>Estados e propriedades

Microsoft Active Accessibility elementos de suporte a um conjunto comum de propriedades. Algumas propriedades, como accState, devem descrever condições diferentes, dependendo da função de elemento. Os servidores devem implementar todos os métodos [**de IAccessible**](/windows/desktop/api/oleacc/nn-oleacc-iaccessible) que retornam uma propriedade, mesmo as propriedades que não são relevantes para o elemento.

Automação da Interface do Usuário define propriedades adicionais, algumas das quais correspondem aos estados em Microsoft Active Accessibility. Algumas propriedades são comuns a todos os elementos, mas outras são específicas para controlar tipos e padrões de controle. Um Automação da Interface do Usuário provedor não precisa implementar propriedades irrelevantes, mas pode retornar um valor nulo para todas as propriedades que ele não dá suporte. O Automação da Interface do Usuário principal pode obter algumas propriedades do provedor de janela padrão e elas são acumuladas com propriedades explicitamente implementadas pelo provedor.

Além de dar suporte a muitas outras propriedades, Automação da Interface do Usuário melhor desempenho permitindo que as propriedades sejam armazenadas em cache.

## <a name="events"></a>Eventos

Ao contrário Microsoft Active Accessibility, o mecanismo de evento no Automação da Interface do Usuário não depende do roteamento de eventos Windows, que está intimamente vinculado a alças de janela e não exige que o aplicativo cliente configurar ganchos. As assinaturas de eventos podem ser ajustadas a determinadas partes da árvore, não apenas a eventos específicos. Os provedores também podem ajustar os eventos de adoção, mantendo o controle de quais eventos estão sendo escutados.

Também é mais fácil para os clientes recuperarem os elementos que aumentem eventos porque eles são passados diretamente para o retorno de chamada de evento. As propriedades do elemento são pré-buscadas automaticamente, se uma solicitação de cache foi fornecida quando o cliente assinou o evento.

## <a name="accessing-active-accessibility-properties-and-objects-from-ui-automation"></a>Acessando Acessibilidade Ativa propriedades e objetos de Automação da Interface do Usuário

Um recurso importante de Automação da Interface do Usuário que não está disponível no Microsoft Active Accessibility é a capacidade de buscar várias propriedades com uma única operação entre processos.

Os clientes Microsoft Active Accessibility existentes podem aproveitar essa capacidade usando a interface [**IUIAutomationLegacyIAccessiblePattern.**](/windows/desktop/api/UIAutomationClient/nn-uiautomationclient-iuiautomationlegacyiaccessiblepattern) Essa interface representa um *padrão de controle* que expõe Microsoft Active Accessibility e métodos em elementos da interface do usuário. Ao recuperar elementos, um aplicativo pode solicitar que esse padrão de controle e suas propriedades sejam armazenados em cache.

[**IUIAutomationLegacyIAccessiblePattern**](/windows/desktop/api/UIAutomationClient/nn-uiautomationclient-iuiautomationlegacyiaccessiblepattern) também permite que os clientes obtenham Microsoft Active Accessibility propriedades de elementos que não têm suporte nativo para [**IAccessible.**](/windows/desktop/api/oleacc/nn-oleacc-iaccessible)

As alterações nas propriedades de [**um IUIAutomationLegacyIAccessiblePattern**](/windows/desktop/api/UIAutomationClient/nn-uiautomationclient-iuiautomationlegacyiaccessiblepattern) não acionam Automação da Interface do Usuário eventos.
